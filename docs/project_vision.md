# Serving Agent 项目愿景

## 1. 问题陈述

### 1.1 当前痛点

**现状问题**：现有 LLM 推理栈在 Ascend NPU 上的部署存在显著障碍：

| 维度 | 问题描述 | 具体表现 |
|------|---------|---------|
| **复杂度高** | 技术栈层次多，依赖关系复杂 | vLLM-Ascend + torch-npu + CANN 版本严格匹配，环境配置耗时 |
| **代码量大** | 现有框架代码规模庞大 | vLLM > 10 万行代码，难以理解和调试 |
| **灵活性差** | 通用框架难以适配多样化需求 | 不同场景需要不同特性组合（单机/多机、KV Cache 策略、批处理方式） |
| **应用闭环缺口** | PyPTO 优势尚未应用价值最大化 | 算子生成能力强，但缺少将算子组装成可部署推理服务的完整链路 |

### 1.2 核心问题

**PyPTO 的定位与机遇**：
- 🎯 **核心定位**：提供统一的 Tile 级抽象，生成高性能算子
- 🌟 **生态机遇**：需要一个应用组装层将算子组合成完整推理应用
- 🚀 **协同价值**：PyPTO 专注于算子生成，Serving Agent 负责应用组装，共同实现端到端解决方案

**开发的困境**：
- 每次新的部署需求（硬件配置、模型规格、并行策略）都需要大量重复开发
- AI 难以理解大规模代码库，无法有效参与代码生成和优化
- 从模型到可部署服务之间存在巨大的"工程鸿沟"

## 2. 项目愿景

### 2.1 定位

**Serving Agent** 是一个**配置驱动的代码生成工具**，连接 PyPTO 的算子生成能力与实际的 LLM 推理部署需求。

### 2.2 与 PyPTO 的协同关系

```

│                     PyPTO 生态系统                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   ┌──────────────┐         ┌──────────────┐               │
│   │  PyPTO 核心   │  ┌───▶  │ Serving Agent│               │
│   │  算子生成能力 │  │      │  应用组装器   │               │
│   └──────────────┘  │      └──────────────┘               │
│          │         │              │                        │
│          ▼         │              ▼                        │
│   ┌──────────────┐ │      ┌──────────────┐               │
│   │  CCE 内核    │ │      │  完整推理服务  │               │

│   └──────────────┘ │      └──────────────┘               │
│          ▲         │              ▲                        │
│          │         │              │                        │
│   ┌──────┴──────┐ │      ┌───────┴───────┐              │
│   │ 算子级能力   │ │      │  应用级能力    │              │

```

**协同价值**：

| PyPTO 贡献 | Serving Agent 贡献 | 结合效果 |
|-----------|-------------------|---------|
| Tile 级算子抽象 | 服务级架构组装 | 从算子到应用的完整链路 |
| 高性能内核生成 | 多样化部署配置 | 一套算子适配多种场景 |
| 统一计算接口 | 并行策略实现 | 单机/多机透明切换 |
| 底层优化能力 | AI 友好框架 | AI 可参与的代码生成 |

### 2.3 预期效果

**成功后的样子**：

#### 场景 1：快速部署新模型
```bash
# 用户只需提供配置
$ cat config.toml
[model]
name = "qwen3"
variant = "8b"
weights_path = "/data/models/qwen3-8b"

[hardware]
backend_type = "pypto"
npus_per_node = 8
nodes = 2

[parallel]
tensor_parallel_size = 4
pipeline_parallel_size = 2

# 一键生成推理服务
$ serving-agent generate --config config.toml --output ./qwen3_server
✓ Generated project: ./qwen3_server (4,821 lines)
✓ Compilation time: 45s
✓ Ready to deploy

# 直接编译运行
$ cd qwen3_server && cargo run --release
🚀 Server ready: http://0.0.0.0:8080
```

```bash
# 从单机改为多机分布式

[parallel]

pipeline_parallel_size = 1  # 从 2 改为 1

# Serving Agent 自动：
# ✓ 选择合适的通信模板
# ✓ 生成分布式协调代码
# ✓ 适配多机启动脚本
```

## 5. 范围界定

### 5.1 项目核心范围

- ✅ 配置解析和验证
- ✅ 基于 Jinja2 的代码生成模板
- ✅ Rust 推理服务框架组装
- ✅ PyPTO 算子集成
- ✅ 单机多卡、多机分布式支持

### 5.2 外部能力诉求与协作边界

**Serving Agent 依赖的外部能力**：

| 能力领域 | 负责团队/组件 | 说明 |
|---------|-------------|------|
| **PyPTO 算子生成** | PyPTO 团队 | 调用 PyPTO API 获取生成的算子内核 |
| **并行策略算法** | 算法团队 | TP/PP 等并行策略的算法实现 |
| **模型量化算法** | 专门团队 | INT8/INT4 等量化算法研究 |
| **通信原语实现** | simpler, pypto_runtime_distributed | 单机多卡、多机通信基础能力 |
| **Ascend NPU 驱动** | CANN | 硬件抽象和驱动支持 |
| **生产级监控运维** | 后续迭代 | 监控、日志、告警等运维工具 |

## 6. 设计原则

### 6.1 简单性优先
- 模板保持简单，避免过度抽象
- 生成代码清晰易懂，避免黑盒
- 配置直观，最小化学习成本

### 6.2 AI 友好
- 控制代码规模，确保 AI 可理解
- 模块职责清晰，便于 AI 修改
- 提供完整的上下文信息

### 6.3 可扩展性
- 模板化设计，易于添加新特性
- 插件化架构，支持新后端
- 版本兼容，支持平滑升级

### 6.4 实用主义
- 不追求完美，优先解决核心问题
- 快速迭代，根据反馈调整
- 保持代码库健康，及时重构

---

## 7. 风险与挑战

| 风险 | 影响 | 缓解策略 |
|------|------|---------|
| PyPTO API 不稳定 | 生成代码可能失效 | 版本锁定 + 抽象层隔离 |
| 模板维护成本高 | 特性组合爆炸 | 保持模板简单，减少组合 |
| 生成代码性能不达标 | 用户不愿意使用 | 提供性能基准，支持优化 |
| 多机分布式复杂性 | 开发难度大 | 分阶段实现，先单机后多机 |

---

## 8. 项目定位总结

**Serving Agent 不是**：
- ❌ 新的推理框架
- ❌ PyPTO 的替代品
- ❌ 通用代码生成工具

**Serving Agent 是**：
- ✅ PyPTO 应用闭环的完成者
- ✅ 配置驱动的服务组装工具
- ✅ AI 友好的代码生成框架
- ✅ 连接算子能力与部署需求的桥梁
